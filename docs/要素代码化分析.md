# xAgt 要素代码化分析

> "文档代码化"——把散落在 prompt 字符串里的设计规则和重复要素，用类型/函数/schema 明确定义，而不是藏在自然语言里。

## 核心观点

- **对话记忆本身已经是代码化的**。`SessionArchiver` 为每场对话生成了 ID + 标题（如 `S001-项目初始化.json`），提示词中可以直接引用 `{recentSessions}` 作为上下文。这证明了"要素代码化"的可行性——提示词不需要写死内容，而是引用代码生成的变量。
- 目标是 **"能抽尽抽"**。凡是跨角色重复的、可独立管理的、不依赖特定 Agent 身份的设计要素，都应该抽出来，不放 prompt 里。

---

## 可抽取层次总览

| # | 抽什么 | 抽成什么 | 当前位置 | 影响范围 |
|---|--------|---------|---------|---------|
| 1 | **Agent 5 维度** | `AgentDefinition` 类型 + 渲染引擎 | 各 agent 的 prompt 字符串 | 全部 5 个 agent |
| 2 | **协作协议** | `PROTOCOLS` 对象 | Vox prompt `<Protocol>` | Vox 调度 |
| 3 | **审查清单** | `JUDGE_CHECKLIST` 数组 | Judge prompt 工作方式 | Judge 行为 |
| 4 | **分级验证** | `VERIFY_STRATEGY` 对象 | Fixer prompt 工作方式 | Fixer 行为 |
| 5 | **输出拦截** | `OUTPUT_RULES` 配置 | index.ts + system-transform.ts | Vox 输出 |
| 6 | **核心红线** | `REDLINES` 数组 | Vox prompt `<Constitution>` | Vox 底线 |
| 7 | **危险命令** | `DANGER_RULES`（已部分代码化） | gateway/policy.ts | Fixer bash |

## 第 1 层：Agent 5 维度模板

### 目标形态

```typescript
interface AgentDefinition {
  name: string
  role: string       // 定位 — 你是谁
  cap: string[]      // 能力 — 你能用什么
  rules: string[]    // 铁律 — 绝对不能做什么
  process: string    // 流程 — 怎么做
  output: string     // 输出 — 怎么汇报
}
```

### 现状

5 个 agent 的 prompt 都直接在字符串中手写，结构不统一：
- Vox 用英文 XML 标签（`<Constitution>`, `<Capabilities>`, `<Protocol>`）
- 子代理用中文 Markdown 标题（`## 你的定位`, `## 你的能力`）
- Smith 缺"流程"维度

### 抽出来后

每个 agent 的创建函数返回 `AgentDefinition`，由统一的渲染引擎生成 prompt 字符串。改 Vox 的规则 = 改 `rules` 数组的一个元素，不用在 500 字里找位置。

## 第 2 层：协作协议

### 目标形态

```typescript
type AgentProtocol = {
  target: string      // 子代理名
  fields: string[]    // 必须传的字段
  template: string    // 格式模板
}

const PROTOCOLS: Record<string, AgentProtocol> = {
  lynx: {
    target: "@lynx",
    fields: ["目标", "线索", "期望"],
    template: "【目标】{goal} | 【线索】{clue} | 【期望】{expected}",
  },
  fixer: {
    target: "@fixer",
    fields: ["文件", "操作", "验收"],
    template: "【文件】{file} | 【操作】{op} | 【验收】{verify}",
  },
  judge: {
    target: "@judge",
    fields: ["审查", "重点", "标准"],
    template: "【审查】{scope} | 【重点】{focus} | 【标准】{passCondition}",
  },
}
```

### 现状

Vox 的 prompt 里用 Markdown 固定写死了三个子代理的指令模板。如果以后加了新子代理或改了模板，需要翻 Vox 的 prompt 字符串。

### 抽出来后

Vox 的 prompt 只需写"按照协作协议给子代理发指令"，具体模板由 `PROTOCOLS` 对象生成。加新子代理 = 在 `PROTOCOLS` 里加一条，Vox 自然就能用。

## 第 3 层：审查清单

### 目标形态

```typescript
type CheckItem = {
  id: string
  label: string
  detail: string
}

const JUDGE_CHECKLIST: CheckItem[] = [
  { id: "A", label: "自动检查", detail: "编译是否通过？测试是否通过？Fixer是否提供验证凭证？" },
  { id: "B", label: "日志规范", detail: "日志前缀/级别/中文/超长/脱敏是否符合logrule？" },
  { id: "C", label: "UI规范", detail: "组件样式是否源码层修改？是否用 @theme 变量？" },
  { id: "D", label: "越权检查", detail: "Fixer是否改了未指定的文件/配置/依赖？" },
  { id: "E", label: "结构检查", detail: "修改是否在正确模块？有无循环依赖？" },
]
```

### 现状

Judge 的 prompt 里用 Markdown 罗列了 5 项审查清单，带 □ 标记。新增审查项需要改 Judge 的 prompt。

### 抽出来后

Judge 的 prompt 只说"按审查清单逐项检查"。清单作为结构化数据在每次审查任务中注入。加新检查项 = 在数组里 push 一条，Judge 自然遵循。

## 第 4 层：分级验证策略

### 目标形态

```typescript
type VerifyLevel = "trivial" | "simple" | "complex"

const VERIFY_STRATEGY: Record<VerifyLevel, {
  label: string
  steps: string[]
  mustNot: string[]
}> = {
  trivial: {
    label: "琐碎修改（注释/typo/重命名）",
    steps: ["tsc --noEmit"],
    mustNot: ["不要跑全量测试"],
  },
  simple: {
    label: "简单修改（改默认值/调参数）",
    steps: ["tsc --noEmit", "bun test 直接相关的测试"],
    mustNot: ["不要跑全量测试"],
  },
  complex: {
    label: "复杂修改（新功能/重构）",
    steps: ["先写测试", "实现代码", "跑相关测试", "验证日志"],
    mustNot: [],
  },
}
```

### 现状

Fixer 的 prompt 里写了三级验证的文字描述。如果验证策略需要调整（比如"琐碎修改也跑 lint"），需要改 Fixer 的 prompt。

### 抽出来后

Fixer 的 prompt 只说"按分级验证策略执行"。具体步骤是可配置的。

## 第 5 层：输出拦截规则

### 目标形态

```typescript
const OUTPUT_RULES = {
  codePatterns: {
    codeBlock: /```[\s\S]*```/,
    inlineCode: /`[^`]{10,}`/,
  },
  exemptions: [
    { if: "含 task() 调用", pattern: /task\s*\(/ },
  ],
  blockMessage: "⛔ 自动拦截：你的回复包含代码但没有调度子代理...",
}
```

### 现状

两处逻辑共同完成输出拦截：
1. `system-transform.ts`：在 Vox 的 system prompt 里注入"输出自检规则"
2. `index.ts`：运行时正则检测，匹配到违规就替换

两个地方的行为是一致的，但定义分散。改了拦截规则需要改两处。

### 抽出来后

单一定义源。system-transform 把规则转成自然语言注入 prompt，index.ts 用同一份规则做运行时检测。改规则只改一处。

## 第 6 层：核心红线

### 目标形态

```typescript
const REDLINES = [
  { id: "judge-gate",    text: "Fixer 完成后必须派 Judge 审查，通过才算完成" },
  { id: "fail-limit",    text: "子代理连续失败 3 次，暂停上报" },
  { id: "think-limit",   text: "思考 3 轮无结论，申请指导，不要死循环" },
  { id: "no-split",      text: "一次 task() 能解决的不要拆两次" },
  { id: "no-overstep",   text: "不替子代理想代码细节 — 越权" },
]
```

### 现状

Vox 的 prompt `<Constitution>` 里写死了 5 条红线。

### 抽出来后

Vox 的 prompt 说"遵守核心红线"。红线定义独立管理。加红线 = push 一条。

## 第 7 层：危险命令

### 目标形态

```typescript
// 当前 gateway/policy.ts 已经有 DangerRule[] 结构
// 但它是嵌在 DEFAULT_POLICY.agents.fixer.dangerRules 里的
// 建议提升为独立的顶级常量
export const DANGER_RULES: DangerRule[] = [
  { pattern: /\brm\s+-rf\b/i, reason: "禁止递归删除（rm -rf）" },
  { pattern: /\bnpm\s+publish\b/i, reason: "禁止 npm 发布操作" },
  // ... 共 16 条
]
```

### 现状

已经半代码化了——`DangerRule` 类型 + `DEFAULT_POLICY` 对象。但 16 条规则嵌在 Fixer 的策略配置里，没有独立命名导出。

### 抽出来后

独立常量，可以被多处引用（如 Smith 审查时可以对照检查）。

## 推进建议

按影响面从大到小，做一层、验证一层：

| 顺序 | 层 | 理由 |
|------|-----|------|
| 1️⃣ | Agent 5 维度模板 | 是基础抽象，后续各层都依赖它 |
| 2️⃣ | 核心红线 | Vox 的底线，简单直接 |
| 3️⃣ | 协作协议 | Vox 调度子代理的核心 |
| 4️⃣ | 分级验证 | Fixer 的行为定义 |
| 5️⃣ | 审查清单 | Judge 的行为定义 |
| 6️⃣ | 输出拦截 | 跨两处逻辑，统一管理 |
| 7️⃣ | 危险命令 | 已经半代码化，提升即可 |
