**文件结构：**
- `src/cost/cache.ts` — 工具调用结果缓存（Lynx 的 grep/glob/webfetch、Judge 审查结果）
- `src/cost/router.ts` — 模型路由逻辑（利用现有 `AgentConfig.model` 字段）

**缓存策略：**
- 缓存对象：只读 Agent 的工具调用结果（幂等操作）
- key 格式：`{action}:{inputHash}`（如 `grep:{pattern}:{path}`）
- TTL：grep/glob 30s，webfetch 60s，Judge 审查 5min
- 淘汰：LRU，最大 100 条
- 失效触发：`tool.execute.before` 检测到 `edit`/`write` 时清除项目缓存

**模型路由：**

### M5 断点续传 — 设计方案

**文件结构：**
- `src/hooks/task-state-persistence.ts` — `TaskStatePersistence` 类（load/save/clear）
- `src/index.ts: system.transform` — 注入恢复指令

**存储格式：** `.xagt/task_state.json`
- 包含：`activeTask`（sandboxRef / instruction / completedSteps / nextStepIndex）
- 关联 M3：`sandboxRef.branchName` 精确到 Git 分支

**恢复流程：**
1. `system.transform` 检测 `.xagt/task_state.json`
2. 检查 `sandboxRef.branchName` 是否存在
3. 分支存在 → 注入恢复指令给 Fixer（"已完成 N/M 步，从第 N+1 步继续"）
4. 分支不存在 → 提示 Vox 重新分派

**推荐方案 B**（代码级状态管理），理由：断点续传是保命功能，不能依赖 LLM 自觉检查文件。
