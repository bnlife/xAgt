# E2E 测试方案：让不懂代码的人也能看懂

> 基于 OpenCode 官方推荐和 oh-my-opencode-slim 项目实践
> 调研日期：2026-07-03

---

## 一、为什么需要 E2E 测试？

**E2E（端到端）测试** 模拟真实用户的操作流程，验证整个系统是否正常工作。

```
用户视角：打开终端 → 输入命令 → 看到回复 → 继续输入
E2E测试：模拟上述完整流程，验证每一步是否正确
```

**没有 E2E 测试的问题**：
- 单元测试只能验证单个函数，无法验证用户真实体验
- 集成测试只能验证组件协作，无法验证完整流程
- 手动测试费时费力，且容易遗漏

## 二、两种 E2E 测试方案

### 方案一：YAML 自然语言测试（推荐，最简单）

**适合人群**：产品经理、测试工程师、业务分析师、任何不懂代码的人

**核心思路**：用自然语言写测试步骤，AI 自动执行

#### 安装

```bash
npm install -g opencode-test
opencode-test init
```

#### 编写测试

创建文件 `tests/login.yml`：

```yaml
# 测试：用户登录功能
tags: [smoke, login]
steps:
  - "打开 OpenCode 界面"
  - "在输入框输入 /interview 我要做一个电商网站"
  - "等待 Agent 回复问题"
  - "在输入框输入回答：Web平台"
  - "验证页面显示访谈状态"
```

**每一行都是人话，不需要写代码。**

#### 运行测试

```bash
opencode-test run
```

#### 查看报告

自动生成 HTML 报告，包含：
- ✅ 测试通过/失败状态
- ✅ 每个步骤的截图
- ✅ 失败时的视频录制
- ✅ 执行时间线

### 方案二：TypeScript 测试（更精确）

**适合人群**：开发者，需要精确控制测试逻辑

**核心思路**：用代码模拟用户操作，验证系统行为

#### 测试文件结构

```
tests/
├── unit/                    # 单元测试（测试单个函数）
│   ├── config.test.ts       # 配置解析测试
│   └── utils.test.ts        # 工具函数测试
├── integration/             # 集成测试（测试多个组件协作）
│   ├── agent-registration.test.ts  # Agent 注册测试
│   └── hook-execution.test.ts      # Hook 执行测试
└── e2e/                     # E2E 测试（模拟完整用户流程）
    ├── interview-flow.test.ts      # 访谈流程测试
    ├── task-delegation.test.ts     # 任务委派测试
    └── error-recovery.test.ts      # 错误恢复测试
```

#### 测试示例（带详细注释）

```typescript
import { describe, it, expect } from 'bun:test'

// =============================================
// 测试：用户使用 /interview 命令创建项目
// 模拟真实用户操作：输入命令 → 看到回复 → 回答问题
// =============================================
describe('用户使用 /interview 命令创建项目', () => {

  // 测试1：用户输入命令后，系统应该开始访谈
  it('用户输入 /interview 命令后，系统应该开始访谈', async () => {
    // 1. 准备：创建模拟的插件上下文（就像准备一个测试环境）
    const mockContext = createMockContext()
    
    // 2. 模拟用户输入：用户在终端输入了 "/interview 我要做一个电商网站"
    const userInput = {
      command: 'interview',
      sessionID: 'test-session-123',
      arguments: '我要做一个电商网站'  // ← 用户的输入
    }
    
    // 3. 执行：调用插件的命令处理函数
    const output = { parts: [] }
    await plugin['command.execute.before'](userInput, output)
    
    // 4. 验证：系统应该返回什么？
    expect(output.parts.length).toBe(1)  // 应该有一条回复
    expect(output.parts[0].text).toContain('电商网站')  // 回复应该包含用户输入的内容
  })

  // 测试2：Agent 回复问题后，用户提交答案应该更新状态
  it('Agent 回复问题后，用户提交答案应该更新状态', async () => {
    // 1. 准备
    const mockContext = createMockContext()
    
    // 2. 模拟 Agent 回复（就像 AI 给用户发了一条消息）
    const agentReply = {
      info: { role: 'assistant' },
      parts: [{
        type: 'text',
        text: '你想做什么平台？\n<interview_state>\n{"questions":[{"id":"q-1","question":"平台选择","options":["Web","Mobile"]}]}\n</interview_state>'
      }]
    }
    
    // 3. 模拟用户回答问题
    const userAnswer = { questionId: 'q-1', answer: 'Web' }
    
    // 4. 执行：提交答案
    await interviewService.submitAnswers('interview-123', [userAnswer])
    
    // 5. 验证：状态应该更新
    const state = await interviewService.getState('interview-123')
    expect(state.status).toBe('in_progress')
  })
})
```

## 三、oh-my-opencode-slim 的测试实践

### 3.1 测试框架

使用 **Bun 内置测试框架**（`bun:test`）：

```bash
bun test                    # 运行所有测试
bun test -t "test-pattern"  # 按名称过滤
```

### 3.2 测试文件分布

项目共有 **73 个测试文件**，全部使用 `*.test.ts` 命名，测试与源码同目录放置。

| 模块 | 测试文件数 | 说明 |
|------|-----------|------|
| 入口 | 1 | 插件入口测试 |
| Agents | 5 | Agent 注册、Council、自定义 Agent |
| Config | 8 | 配置加载、schema 验证、preset |
| Hooks | 12+ | 后台任务管理、fallback、补丁应用 |
| Tools | 5+ | 预设管理、任务取消、智能抓取 |
| Utils | 8 | 轮询、会话、日志、环境变量 |
| CLI | 10 | 安装、配置、诊断、缓存 |
| Interview | 4 | 访谈服务、管理器、Dashboard |
| Multiplexer | 4 | Tmux/Zellij/Herdr 面板集成 |

### 3.3 核心测试模式

#### 模式1：Mock 插件上下文工厂

```typescript
// 适用于所有需要模拟 OpenCode 插件上下文的测试
function createMockContext(overrides?) {
  return {
    client: {
      session: {
        messages: mock(async () => ({ data: overrides?.messagesData ?? [] })),
        prompt: mock(async () => ({})),
        promptAsync: mock(async () => ({})),
      },
    },
    directory: overrides?.directory ?? '/test/directory',
  } as any;
}
```

#### 模式2：临时目录 + try/finally 清理

```typescript
// 适用于有文件系统副作用的测试
test('test name', async () => {
  const tempDir = await fs.mkdtemp('/tmp/test-');
  try {
    // ... 测试逻辑使用 tempDir ...
  } finally {
    await fs.rm(tempDir, { recursive: true, force: true });
  }
});
```

#### 模式3：真实 HTTP 服务器集成测试

```typescript
// 适用于需要验证完整 HTTP 请求/响应链的测试
const { baseUrl, authToken, dashboard, cleanup } = await startDashboard();
try {
  const response = await fetch(`${baseUrl}/api/something?token=${authToken}`);
  expect(response.status).toBe(200);
} finally {
  cleanup();
}
```

#### 模式4：模拟事件驱动生命周期

```typescript
// 适用于测试事件处理器、钩子系统
await hook['tool.execute.before']({ tool: 'task', sessionID: 'p-1', callID: 'c-1' }, { args: {...} });
await hook['tool.execute.after']({ tool: 'task', sessionID: 'p-1', callID: 'c-1' }, { output: '...' });
await hook.event({ event: { type: 'session.status', properties: { sessionID: 'p-1', status: { type: 'idle' } } } });
```

#### 模式5：Deferred Promise 并发测试

```typescript
// 适用于模拟竞态条件、锁行为等并发场景
function createDeferred<T>() {
  let resolve!: (value: T) => void;
  let reject!: (reason?: unknown) => void;
  const promise = new Promise<T>((res, rej) => {
    resolve = res;
    reject = rej;
  });
  return { promise, resolve, reject };
}

// 使用示例：模拟正在进行中的 spawn
const deferred = createDeferred<{ success: true; paneId: string }>();
mockMultiplexer.spawnPane.mockImplementationOnce(() => deferred.promise);
// ... 执行需要 spawn 完成的逻辑 ...
deferred.resolve({ success: true, paneId: '%new-pane' }); // 手动完成
```

### 3.4 测试可读性分析

**优点**：
- 测试描述清晰：每个 `test()` 的名称都是完整的句子
  ```typescript
  test('creates interview with /interview command')
  test('marks interview as abandoned on session.deleted event')
  test('rejects concurrent submission when first request holds busy lock')
  ```
- 嵌套 describe 提供清晰的上下文层次
  ```typescript
  describe('interview service')
    describe('/interview <idea> command')
      describe('answer submission')
        describe('session interview lifecycle')
  ```
- 辅助函数命名具有表达力：`createMockContext()`、`findFreePort()`、`createDeferred()`

**不足**：
- 没有注释说明测试的"为什么"
- Magic numbers 存在（`setTimeout(r, 100)` 等延迟值没有文档说明）
- Mock 上下文创建代码在多个文件中重复
- 部分测试文件过长（最长的 1675 行）

## 四、如何让不懂代码的人也能看懂测试

### 4.1 命名规范

```typescript
// ❌ 不好的命名
test('test1', ...)
test('interview', ...)

// ✅ 好的命名（完整的句子）
test('用户输入 /interview 命令后，系统应该开始访谈')
test('Agent 回复问题后，用户提交答案应该更新状态')
test('用户取消任务后，后台任务应该被终止')
```

### 4.2 注释规范

```typescript
// =============================================
// 测试场景：用户使用 /interview 命令创建项目
// 
// 用户操作：
//   1. 在终端输入 /interview 我要做一个电商网站
//   2. 系统返回访谈问题
//   3. 用户回答问题
//   4. 系统更新访谈状态
//
// 验证点：
//   - 系统应该回复用户输入的内容
//   - 访谈状态应该更新
// =============================================
```

### 4.3 测试即文档

```yaml
# tests/e2e/interview-flow.yml
# 
# 这个文件描述了一个完整的用户访谈流程
# 任何人都可以读懂，不需要编程知识
#
# 用户故事：
#   小明想做一个电商网站，他打开 OpenCode，输入 /interview 命令
#   AI 开始问他问题，他一一回答
#   最后 AI 生成了一份完整的项目规格文档

name: "用户访谈完整流程"
tags: [e2e, interview]

steps:
  # 第1步：用户启动访谈
  - "打开 OpenCode 终端"
  - "输入 /interview 我要做一个电商网站"
  
  # 第2步：AI 提问，用户回答
  - "等待 AI 提问：你想做什么平台？"
  - "输入回答：Web平台"
  - "等待 AI 提问：目标用户是谁？"
  - "输入回答：普通消费者"
  
  # 第3步：验证结果
  - "验证系统显示访谈完成"
  - "验证生成了项目规格文档"
```

## 五、测试运行和 CI/CD 集成

### 5.1 本地运行

```bash
# 运行所有测试
bun test

# 运行特定层级的测试
bun test tests/unit
bun test tests/integration
bun test tests/e2e

# 按名称过滤
bun test -t "interview"

# 代码质量检查
bun run check:ci
bun run typecheck
```

### 5.2 CI/CD 集成

```yaml
# .github/workflows/e2e.yml
name: E2E Tests
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test
        env:
          BASE_URL: ${{ vars.STAGING_URL }}
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
```

### 5.3 开发工作流

```
修改代码 → bun run check:ci → bun run typecheck → bun test → commit
```

## 六、两种方案对比

| 特性 | YAML 自然语言测试 | TypeScript 测试 |
|------|------------------|----------------|
| **学习难度** | ⭐ 极低 | ⭐⭐⭐ 中等 |
| **可读性** | ⭐⭐⭐ 非常高 | ⭐⭐ 需要基础 |
| **灵活性** | ⭐⭐ 中等 | ⭐⭐⭐ 非常高 |
| **执行速度** | ⭐ 较慢（需要AI） | ⭐⭐⭐ 快 |
| **维护成本** | ⭐⭐ 中等 | ⭐⭐⭐ 较高 |
| **适合人群** | 产品经理、测试、BA | 开发者 |
| **适合场景** | 核心功能验证 | 边界情况、错误恢复 |

---

## 七、建议

**对于多 agent 插件项目，建议：**

1. **先用 YAML 自然语言测试** 验证核心功能
   - 用户能否正常启动访谈
   - Agent 能否正确回复
   - 任务委派是否工作

2. **再用 TypeScript 测试** 覆盖边界情况
   - 并发任务处理
   - 错误恢复
   - 状态持久化

3. **最终目标**：让测试成为文档
   - YAML 测试 = 用户手册
   - TypeScript 测试 = 技术规格
