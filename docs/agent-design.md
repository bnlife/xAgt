# Agent 设计规范

## 设计维度

每个 Agent 的 prompt 必须覆盖以下 **6 个必选维度**：

| # | 维度 | 含义 | 示例 |
|---|------|------|------|
| 1 | **定位** | 一句话说清你是谁，在整个系统中的位置 | "你是总指挥，只调度不干活" |
| 2 | **职责** | 你负责做什么 | "精准理解意图、调度子代理、跟踪进度、整合结果" |
| 3 | **能力** | 你有什么工具、什么技能，边界在哪 | "task/read/skill/todowrite，没有 write/edit/bash" |
| 4 | **规则** | 绝对不能做的事（铁律）+ 必须遵守的底线 | "不亲自写代码""连续失败3次暂停" |
| 5 | **流程** | 收到任务后做什么、怎么做、步骤是什么 | "理解需求→调研→设计→规划→实现→测试→交付" |
| 6 | **输出** | 任务完成后怎么汇报、什么格式 | "结论先于依据""直接回答不寒暄" |

### 可选维度（角色特定）

| 维度 | 适用角色 | 说明 |
|------|---------|------|
| **协作协议** | Vox | 如何给子代理发指令的模板 |
| **前置准备** | Judge | 审前必须加载的规范 |
| **环境约束** | 按需 | 网络限制、系统限制等 |

## 现状问题

1. **结构不统一**：同一个"定位"在 Vox 叫 `<Constitution>`，在 Lynx 叫 `## 你的定位`，在 Judge 叫 `## 你的定位`
2. **维度混乱**：Lynx 的能力和工具分了两个章节（`## 你的能力` + `## 你的 MCP 工具`），但本质是同一个维度
3. **Smith 缺流程**：唯一没有"怎么做"的 Agent——它不知道该做什么步骤
4. **表述风格不一致**：Vox 用英文 XML 标签，子代理用中文 Markdown 标题
